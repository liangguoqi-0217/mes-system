\# COMPONENT LIBRARY



> Enterprise UI Component Library



Version: 1.0



\---



\# 1. Purpose（目的）



本规范定义整个产品体系的标准组件。



组件不是视觉元素。



组件是业务交互的基本单元。



所有页面应优先使用标准组件，而不是自行设计新的组件。



组件数量应保持最少。



组件行为应保持一致。



\---



\# 2. Component Principle（组件原则）



组件存在的唯一目的：



帮助用户完成业务。



组件越少。



学习成本越低。



一致性越高。



组件的选择，应由业务决定，而不是由设计决定。



\---



\# 3. Data Table（数据表格）



数据表格是企业软件最重要的组件。



默认支持：



\- 排序

\- 筛选

\- 列宽调整

\- 列显示/隐藏

\- 导出 Excel



以下能力按业务启用：



\- 固定列

\- 多选

\- 行点击查看详情

\- 行内编辑



默认情况下，不启用特殊能力。



\---



\# 4. Search Area（查询区）



查询区默认展开。



支持手动收起。



查询条件较少时，全部展示。



查询条件较多时：



默认展示常用条件。



提供"更多条件"展开。



避免一次展示大量查询项。



\---



\# 5. Button（按钮）



按钮数量由业务复杂度决定。



避免为了统一而限制业务。



默认原则：



主操作突出。



次操作弱化。



危险操作明确标识。



删除操作默认采用"删除标记"。



不鼓励物理删除。



按钮应体现业务含义。



而不是视觉层级。



\---



\# 6. Dialog（弹窗）



弹窗用于：



快速创建。



快速维护。



简单输入。



用户完成保存后。



立即关闭弹窗。



避免在弹窗内完成复杂业务流程。



\---



\# 7. Create / Edit（新增 / 编辑）



新增。



编辑。



查看。



应采用统一页面结构。



区别仅在于：



字段是否允许编辑。



操作按钮是否不同。



避免分别设计三套页面。



\---



\# 8. Tree（树）



Tree 应根据业务决定。



适用于：



组织架构。



设备结构。



分类体系。



区域层级。



BOM。



存在天然层级关系的数据。



不应为了使用 Tree 而使用 Tree。



\---



\# 9. Tabs（标签页）



Tabs 用于组织大量信息。



适用于：



字段较多。



信息可自然分类。



默认可包括：



\- 基本信息

\- 附件

\- 审批记录

\- 操作日志

\- 扩展信息



避免过多 Tabs。



避免信息碎片化。



\---



\# 10. Component Selection（组件选择）



优先选择：



简单组件。



成熟组件。



稳定组件。



避免：



同一页面大量混用不同组件。



例如：



Card。



Collapse。



Drawer。



Popover。



Timeline。



Tree。



Tabs。



若业务无需使用，则不应增加。



\---



\# 11. Component Reuse（组件复用）



所有业务页面。



应优先复用已有组件。



不得因视觉原因重新开发组件。



新增组件必须满足：



\- 存在明确业务价值

\- 可以跨项目复用

\- 不影响整体一致性



\---



\# 12. Component Review Checklist（组件评审）



组件设计完成后，应确认：



□ 是否已有标准组件可以满足需求。



□ 是否由业务驱动组件选择。



□ 是否避免增加新的组件类型。



□ 是否保持组件行为一致。



□ 是否降低用户学习成本。



□ 是否避免复杂组件组合。



优秀的企业软件，不在于拥有更多组件。



而在于使用更少、更稳定、更一致的组件完成更多业务。

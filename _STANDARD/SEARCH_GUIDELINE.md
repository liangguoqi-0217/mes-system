\# SEARCH GUIDELINE



> Search \& Query Guideline



Version: 1.0



\---



\# 1. Purpose（目的）



查询用于帮助用户快速定位业务数据。



查询应围绕业务目标设计。



而不是围绕搜索能力设计。



\---



\# 2. Core Principle（核心原则）



查询应明确、可预测。



用户应知道：



查询什么。



为什么查询。



查询结果来自哪里。



避免模糊、隐式和不可预期的行为。



\---



\# 3. Default State（默认状态）



页面默认不自动查询。



用户主动执行查询后。



系统返回结果。



避免页面初始化即加载大量数据。



\---



\# 4. Search Button（查询按钮）



查询按钮统一放置于：



查询区域右侧。



保持所有页面一致。



支持 Enter 快捷查询。



\---



\# 5. Search Conditions（查询条件）



默认全部展开。



查询条件较多时。



允许折叠。



避免将大量条件长期占据页面空间。



\---



\# 6. Fuzzy Search（模糊查询）



普通文本字段。



默认支持模糊查询。



例如：



供应商名称。



设备名称。



物料名称。



帮助用户快速定位目标。



\---



\# 7. Exact Match（精确匹配）



业务编码默认采用精确匹配。



例如：



订单号。



物料编码。



批号。



设备编号。



避免模糊匹配造成业务歧义。



\---



\# 8. Default Values（默认值）



日期默认保持为空。



下拉框默认：



全部。



避免系统主动限制查询范围。



\---



\# 9. Query Execution（查询执行）



修改查询条件后。



应由用户主动点击查询。



避免自动刷新结果。



保持系统行为可预测。



\---



\# 10. Search History（查询历史）



支持记录最近查询。



推荐保存最近 10 次查询条件。



方便重复业务操作。



\---



\# 11. Saved Query（收藏查询）



是否支持保存查询条件。



由具体业务决定。



避免增加低频功能。



\---



\# 12. Search Scope（查询范围）



查询范围应符合业务目标。



避免默认搜索所有字段。



避免无边界全文搜索。



不同业务对象。



应提供不同查询条件。



查询应服务于业务。



而不是展示搜索能力。



\---



\# 13. Review Checklist（评审清单）



设计完成后，应确认：



□ 默认状态是否清晰。



□ 查询按钮位置是否统一。



□ 是否支持 Enter 查询。



□ 文本是否支持合理模糊查询。



□ 编码是否采用精确匹配。



□ 是否避免自动查询。



□ 查询范围是否符合业务。



优秀的查询。



应让用户明确知道：



系统正在查询什么。



以及为什么得到当前结果。
